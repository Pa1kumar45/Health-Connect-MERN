================================================================================
VIDEO CALL ISSUE RESOLUTION - Health Connect MERN Application
================================================================================

PROBLEM SUMMARY:
- Video calls worked when User initiated call to Friend (User → Friend)
- Video calls FAILED when Friend initiated call to User (Friend → User)
- Issue: Remote video not displaying despite WebRTC connection being established
- Both users on same network (hotspot), using Chrome browser

SYMPTOMS OBSERVED:
1. WebRTC connection established successfully (ICE candidates exchanged)
2. Both sides had MediaStreamTracks (2 tracks: audio + video)
3. Only local video showed in small corner, remote video area was black
4. Console showed: "Waiting for peer connection... retry 1-9"
5. Error: "No peer connection available after waiting"
6. Error: "ICE connection state: disconnected"
7. On answerer side: "The play() request was interrupted by a new load request"

ROOT CAUSES IDENTIFIED:

1. STALE CLOSURE ISSUE (Primary Issue):
   - useEffect with [socket, isInCall] dependencies caused handlers to re-register
   - When answerCall() called setIsInCall(true), useEffect re-ran
   - Socket handlers were unregistered and re-registered with NEW closures
   - New closures captured peerConnectionRef.current at re-registration time (null)
   - When call-answered event fired, handler couldn't find peer connection

2. VIDEO ELEMENT INTERRUPTION ISSUE:
   - Remote video useEffect had dependencies [remoteStream, callStatus]
   - Triggered twice: when remoteStream set AND when callStatus changed to 'connected'
   - Each srcObject assignment interrupted previous play() attempt
   - Result: "AbortError: play() request was interrupted by new load request"

3. CLOSURE VARIABLE ACCESS:
   - Handlers inside useEffect captured state variables (callerId) at registration time
   - When state changed, handlers still had old values (stale closure)
   - ICE candidate handler couldn't send to correct peer

SOLUTIONS IMPLEMENTED:

FIX 1: Remove isInCall from useEffect dependencies
- Changed: }, [socket, isInCall]); 
- To: }, [socket]); 
- Prevents handlers from re-registering when call state changes

FIX 2: Use refs instead of state for handler access
- Added callerIdRef to store latest callerId value
- Sync ref with state: useEffect(() => { callerIdRef.current = callerId; }, [callerId])
- Updated handlers to use callerIdRef.current instead of captured callerId
- Ensures handlers always access latest value, not stale closure value

FIX 3: Prevent duplicate video srcObject assignments
- Added check: if (remoteVideoRef.current.srcObject !== remoteStream)
- Only sets srcObject if it's different from current value
- Prevents interruption of ongoing video playback

FIX 4: Enhanced debugging
- Added emoji-based logging (🔧 🔔 ✅ ❌) for easy log parsing
- Logged when handlers are registered/cleaned up
- Tracked peerConnectionRef state at critical points
- Helped identify timing and closure issues

KEY LEARNINGS:

1. REACT CLOSURES: Event handlers inside useEffect capture variables at registration
   time. If useEffect re-runs, handlers are recreated with new closures.

2. REFS FOR MUTABLE VALUES: Use refs (not state) when handlers need latest values.
   Refs persist across renders without causing re-renders.

3. USEEFFECT DEPENDENCIES: Carefully choose dependencies. Including state that 
   changes during handler execution causes re-registration with stale closures.

4. WEBRTC TIMING: Peer connection must persist from creation through answer receipt.
   Any cleanup or re-initialization breaks the connection flow.

5. VIDEO ELEMENT BEHAVIOR: Setting srcObject multiple times interrupts playback.
   Always check if srcObject is already set before reassigning.

FINAL WORKING FLOW:

Friend → User Call:
1. Friend calls startCall() → creates peerConnection → stores in ref
2. Socket emits call-user with offer
3. User receives call-made event → handleCallOffer runs → sets ringing state
4. User clicks "Answer Call" → answerCall() runs → creates peer connection
5. User sends answer-call with answer SDP
6. Friend receives call-answered event → handleCallAnswered runs
7. Handler accesses peerConnectionRef.current directly (no waiting needed)
8. Sets remote description → exchanges ICE candidates → connection established
9. ontrack event fires → setRemoteStream() → video elements render both streams

RESULT: Both directions (User → Friend and Friend → User) now work correctly.
